The Problem
Voice agents make decisions.
Most can't explain them.

When a voice AI tells a patient their medication is approved, or tells a customer their loan is denied what actually decided that? Without Manthan, nobody knows.

Without Manthan

  • LLM reasoning leaks into decision logic — non-reproducible
  • No audit trail for regulatory review (HIPAA, SEBI, RBI, SOX)
  • Voice agent invents explanations post-hoc
  • Inconsistent outcomes for identical inputs
  • Fail-open: uncertainty defaults to ALLOW
  • No versioned schema past decisions unprovable

With Manthan

  • Rules evaluated deterministically — zero LLM involvement
  • Every decision is a signed, versioned, replayable token
  • Explanation derived only from matched rule — no invention
  • Same input always produces same output, provably
  • Fail-closed: any uncertainty or error → BLOCK
  • Schema + rule snapshot guarantees historical reproducibility
Architecture
How a voice interaction
becomes a decision

The voice agent's job is to listen and extract. Manthan's job is to decide. These two paths never mix.

01

Voice Agent Listens

STT + NLU extracts intent, entities, and context from the caller's speech. AI operates here in comprehension only, never in evaluation.

AI ALLOWED
02

Signals Computed

Deterministic signals are computed outside Manthan patient eligibility score, account standing, risk tier — then passed as structured inputs. Signal logic is fixed, versioned, and auditable.

DETERMINISTIC
03

Schema Validation

Decision Input is validated against the versioned schema. Every required field must be present and correctly typed. Any mismatch returns INVALID — no guessing, no defaults.

INVALID → BLOCK
04

Completeness Check

All required fields must carry values. Missing data returns INCOMPLETE — the agent is instructed to re-prompt the caller for the missing context.

INCOMPLETE → HOLD
05

Rule Evaluation

Manthan evaluates the Decision Input against the versioned rule set. No AI. No inference. Pure rule matching. Returns ALLOW, BLOCK, or ESCALATE with the matched rule ID.

DETERMINISTIC
06

Signed Token Issued

A signed decision token is produced — proof_hash + signature + rule_version + event_id. The voice agent may only act on ALLOW tokens. The token is the authorization.

ALLOW ONLY
07

Voice Agent Responds

The agent reads the decision and the derived explanation back to the caller. The explanation is pulled only from the matched rule — the AI cannot alter or embellish it.

AI READS ONLY
High-Stakes Use Cases
Built for conversations
where errors have consequences

Every use case below involves a decision that affects a real person — their health, their money, their rights. Determinism isn't optional.

Healthcare
Medication Authorization

Voice agent verifies prior authorization for prescriptions over the phone. Rules encode payer criteria, formulary tiers, and patient eligibility evaluated identically every time.

  • Drug class × formulary tier → ALLOW / BLOCK / ESCALATE
  • Prior auth window enforced by rule, not agent memory
  • Denial reason derived from matched rule for EOB compliance
  • Pharmacist escalation path hardwired, not inferred
Healthcare
Clinical Triage & Routing

Inbound patient calls are triaged by symptom signals and routed to the appropriate care level ED, urgent care, or GP callback without a clinician on the first hop.

  • Symptom severity score thresholds set by clinical rule, not AI
  • Fail-closed: ambiguous presentation always escalates
  • Red-flag keyword signals computed before rule entry
  • Full trace for post-incident clinical audit
Healthcare
Insurance Claim Adjudication

Voice agent handles first-notice-of-loss calls for health insurance. Manthan adjudicates coverage eligibility in real time — every rule tied to policy version and member plan.

  • Policy version snapshot bound to each decision token
  • Exclusion lists evaluated as deterministic signal inputs
  • Partially complete claims route to human adjuster — never auto-approved
  • Replay-safe: event_id prevents duplicate adjudication
Financial
Loan & Credit Decisioning

A caller applies for a personal loan via voice. Manthan evaluates the application against underwriting rules — credit tier, debt-to-income, employment status in one deterministic pass.

  • Credit score band × DTI ratio → tiered approval decision
  • Adverse action notice generated from matched rule for FCRA
  • No LLM can soften or override a BLOCK decision
  • Every denial is replayable for regulatory review
Financial
Transaction Dispute Resolution

Customer calls to dispute a transaction. Voice agent collects context; Manthan evaluates dispute eligibility rules amount thresholds, merchant category, dispute window.

  • Dispute window enforced by rule — not guessed by agent
  • Chargeback eligibility tied to merchant category code signals
  • High-value disputes auto-escalate by rule, never by AI judgment
  • Signed token required before provisional credit is issued
Financial
KYC & Account Onboarding

Voice-first onboarding for bank accounts. Manthan gates each step — identity verified, watchlist clear, document valid with a decision token required to proceed.

  • Watchlist match → immediate BLOCK, no agent override
  • Document validity signals computed, not inferred by LLM
  • Step-gated: each phase requires prior ALLOW token
  • Full onboarding trace satisfies AML record-keeping rules

Core Guarantees
What Manthan promises.
Contractually, not aspirationally.
▣
Determinism

Same Decision Input always produces the same Decision Output. No variance, no mood, no context window drift.

⊘
No AI in evaluation

AI helps structure input and read output. It never touches rule evaluation. The core engine is pure logic.

↯
Fail-closed

On error, uncertainty, invalid schema, or incomplete data — the answer is always BLOCK. Never ALLOW by default.

⬡
Signed tokens

Every decision produces a cryptographically signed token. Execution is gated on token validity. Replay is prevented by event_id.

⟳
Reproducibility

Past decisions can be replayed exactly using the original schema + rule snapshot. History is immutable and auditable.

≡
Versioned evolution

No hidden defaults. No silent changes. Every schema or rule change requires a version bump. No retroactive mutations.
