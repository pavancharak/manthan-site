Decision pipeline
01
Validation
Input is checked against the versioned schema. Any mismatch terminates immediately.
→ INVALID
02
Completeness
All required fields must be present. A partial input never reaches rule evaluation.
→ INCOMPLETE
03
Rule evaluation
Input is evaluated against the versioned rule set. No defaults. No interpolation.
→ ALLOW / BLOCK / ESCALATE
04
Explanation
The explanation is derived only from the matched rule. Nothing is inferred or generated.
→ TRACE BOUND
Fail-closed guarantee — any stage can terminate the pipeline. The system never allows on uncertainty or error.
Architecture

Five layers.
One guarantee.

Each layer has a clear boundary and a single responsibility. Nothing crosses layers without an explicit contract.

Definition
Versioned schema + rule set. Co-constrained — neither can drift independently.
Signal
Computed outside the core. Deterministic only. AI may suggest, never define.
Core
Evaluates decision input against rules. No AI. No state. No mutation.
Token
Signed proof artifact: hash + signature + trace + event_id + rule_version.
Gateway
Verifies token before execution. ALLOW only. Whitelist enforced. Replay blocked.

Hard constraints

What the system
will never do.

×
No change without version increment
Every schema and rule change is explicit, versioned, and committed.
×
No retroactive changes to past decisions
Past decisions remain reproducible with the original schema and rule snapshot.
×
No hidden defaults introduced
Every field is explicit. Nothing is assumed. Nothing falls through silently.
×
No bypass of validation or completeness
The pipeline cannot be shortcutted. Every stage runs in order.
×
No change that breaks determinism
All evolution paths — schema, rules, signals, execution — preserve reproducibility.
×
No self-modification
The system does not learn, adapt, or evolve without explicit human instruction.
